# UI仕様書

## 1. UIコンセプト
tokutoku のUIは、Notionのような自由さよりも、
「何も考えずに入れられる」「見返しやすい」を優先する。

思想:
- 入力は軽く
- 閲覧は視覚的に
- 整理は後から

## 2. 主要画面

### 2.1 ホーム / 一覧画面
構成例:
- 上部: URL追加エリア
- 中央: インラインプレビュー
- 下部またはメイン: 記事カード一覧
- サイドまたは上部: フィルタ

### 2.2 詳細画面 / 詳細パネル
- 記事の詳細表示
- メモ編集
- 状態変更
- タグ編集
- 元記事へのリンク

---

## 3. 記事追加UI

### 3.1 URL入力エリア
表示要素:
- URL入力欄
- 読み込み状態
- 保存ボタン

### 3.2 インラインプレビュー
URL入力後、同一画面上にプレビューカードを表示する

プレビュー表示内容:
- 画像
- タイトル
- description
- サイト名

### 3.3 状態
#### 取得中
- skeletonまたはローディング表示

#### 取得成功
- カード表示
- 保存ボタン活性化

#### 取得失敗
- エラーメッセージ表示
- URLだけで保存可能

---

## 4. 記事一覧UI

### 4.1 表示形式
カード型グリッドまたはカード型リストを採用する

優先事項:
- 画像が見える
- 一覧でパッと判別できる
- ブクログやKindleのように眺められる

### 4.2 カード項目
- サムネイル画像
- タイトル
- description（1〜2行）
- statusバッジ
- タグ
- 更新日時（軽く）

---

## 5. 詳細UI

### 5.1 詳細表示項目
- タイトル
- URL
- サイト名
- description
- note
- tags
- status

### 5.2 編集項目
- note
- status
- tags

### 5.3 操作
- 元記事を開く
- 保存
- 削除

---

## 6. フィルタUI
以下をMVP対象とする

- ステータス別タブまたはチップ
- タグフィルタ
- キーワード検索

---

## 7. UX上の重要要件

### 7.1 入力コスト最小
理想の導線:
1. URL貼る
2. 自動取得
3. 内容確認
4. 保存

### 7.2 画面遷移最小
追加時に別画面へ遷移しない

### 7.3 プレビューの存在意義
「本当に読めているか」をその場で確認できること

### 7.4 Notionとの差別化
- URL追加が軽い
- 画像一覧で見返せる
- 状態管理が標準である
- 初期設計が不要

---

## 8. デザイン方針
- やわらかく軽い印象
- 情報密度は高すぎない
- カードの視認性を重視
- “とくとく貯まる”感を表現できる余白や流れを意識
