# 要件定義書

## 1. 対象ユーザー
主な対象ユーザーは、日常的にWeb記事に触れ、あとで読み返したり考えたりしたい個人ユーザーである。

## 2. 利用シナリオ
### シナリオA: とりあえず保存したい
1. ユーザーが記事URLを見つける
2. tokutoku に URL を貼る
3. タイトル、説明、画像がインラインで表示される
4. 内容が取れていることを確認して保存する
5. デフォルト状態で inbox に入る

### シナリオB: あとで整理したい
1. ユーザーが記事一覧を見る
2. 気になる記事を開く
3. 状態変更、タグ付け、メモ追加を行う

### シナリオC: あとで見返したい
1. タグや状態で絞り込む
2. 画像付きカードから視覚的に探す
3. 記事を開いて再読する

## 3. システム対象範囲
### MVPで対象
- 記事追加
- 記事一覧
- 記事詳細
- 状態管理
- タグ管理
- メモ編集
- OGP/メタ情報取得
- 基本的な絞り込み・並び替え

### MVPで対象外
- クローリング
- レコメンド
- 外部共有
- 外部投稿連携
- コメント自動生成
- AI分類
- チーム利用

## 4. コアオブジェクト
### Article
記事ストックの基本単位

保持する主な情報:
- id
- url
- title
- description
- imageUrl
- siteName
- note
- status
- tags
- createdAt
- updatedAt

## 5. 状態管理
MVPでは記事状態として少なくとも以下を持つ。

- inbox
- reading
- thinking
- done

### 各状態の意味
- inbox: とりあえず入れた
- reading: 読み途中・確認中
- thinking: コメントや整理を考えている
- done: 処理済み、投稿済み、整理済み

## 6. 記事追加導線
記事追加は以下の要件を満たす。

- URL を貼り付けて追加できる
- 別画面遷移なし
- 同一画面内でメタ情報プレビューを確認できる
- プレビュー確認後に保存できる
- OGP取得失敗時も、最低限 URL ベースで保存可能とする

## 7. データ保存方針
MVPでは記事情報は永続保存する。
flowba と異なり、本プロダクトの価値はストック管理にあるため、DB保存を前提とする。
